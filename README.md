# `.hq` — центр координации (headquarters)

`.hq` — **управляющий слой** над всеми персональными репозиториями в `d:\GitHub\Personal\`.
Это «почта (треды) + база знаний + очередь задач + связь с человеком» для агентов, которые
работают в отдельных репозиториях, но должны координироваться между собой и со мной.

Имя с точки (`.hq`) подчёркивает служебный характер и отличает папку от репозиториев.
`.hq` сам по себе — **отдельный git-репозиторий** (история, переносимость, бэкап).

> **Это единственный источник правды (single source of truth).** Все остальные документы
> (`comms/README.md`, `human/README.md`, инструкции-стензы внутри репозиториев и т.д.)
> ссылаются сюда, а не дублируют правила. Меняешь протокол — меняешь **этот** файл.

Документы машиночитаемы: каждый артефакт начинается с YAML-frontmatter, чтобы будущие
агенты-оркестраторы парсили их без эвристик. Человекочитаемое тело идёт после frontmatter.

---

## 0. Кто и зачем сюда пишет

Ты работаешь в одном репозитории, но заметил то, что выходит за его границы:
нужно изменение в соседнем репо, есть проблема/идея, охватывающая несколько проектов,
требуется решение или ручное действие человека. Не держи это в голове и не чини молча
чужой репозиторий — **оставь запись здесь**, по таблице маршрутизации ниже. Позже
агенты-оркестраторы прочитают эти записи, разобьют на задачи агентам конкретных репо
и запустят выполнение.

## 1. Области (scope) — куда что класть

| Scope | Что это | Где живёт |
|---|---|---|
| **single-repo** | идея / задача / знание про ОДИН существующий репо | `projects/<repo>/{ideas,tasks,knowledge}/` |
| **cross-repo** | охватывает ≥2 репо | `ideas/` или `tasks/` |
| **no-repo-yet** | будущий проект без репозитория | `projects/<name>/` со `status: no-repo-yet` в `card.md` |
| **общение** | сообщение → ответ → ответ (треды/деревья) | `comms/` |
| **для человека** | ручные действия и ключевые решения | `human/` |

**Данные всегда централизованы здесь**, внутрь репозиториев данные не кладутся.
В каждом репо лежит только короткая инструкция-указатель (локальная — git-ignore либо
`skip-worktree`), которая ссылается на этот протокол — см. §7.

## 2. Таблица маршрутизации — «заметил X → пиши сюда»

| Заметил | Куда писать |
|---|---|
| Идея / задача / знание про ОДИН существующий репо | `projects/<repo>/{ideas,tasks,knowledge}/` |
| Нужно изменение / проблема / вопрос к ДРУГОМУ репо | новый тред в `comms/` с `to: <repo>` |
| Идея, охватывающая ≥2 репо | `ideas/IDEA-<date>-<slug>.md` |
| Решённое/запускаемое изменение через ≥2 репо | `tasks/TASK-####-<slug>.md` + строка в `tasks/QUEUE.md` |
| Будущий проект, репозитория ещё нет | `projects/<name>/card.md` со `status: no-repo-yet` |
| Знание про связи/границы между репо | `knowledge/` (`dependency-graph.md`, `ownership.md`, `research/`, `howto/`) |
| Нужно ручное действие человека (секрет, branch protection, ревью…) | `human/tasks/HT-####-<slug>.md` |
| Нужно решение/выбор человека | `human/decisions/DEC-####-<slug>.md` |

Структура каждого артефакта — в `_templates/`. **Всегда копируй шаблон**, не пиши с нуля.

## 3. Идентификаторы (ID-схемы)

Стабильные ID нужны, чтобы перекрёстные ссылки переживали перемещение файлов.

| Тип | Формат | Пример |
|---|---|---|
| Тред | `T-<YYYYMMDD>-<slug>` | `T-20260609-pk08-rollout` |
| Сообщение в треде | `<thread-id>#<NN>` | `T-20260609-pk08-rollout#02` |
| Идея | `IDEA-<YYYYMMDD>-<slug>` | `IDEA-20260609-shared-cli-cache` |
| Задача | `TASK-####` (единый глобальный счётчик) | `TASK-0007` |
| Решение человека | `DEC-####` | `DEC-0003` |
| Ручная задача человека | `HT-####` | `HT-0005` |

- **Дата-префикс** в именах файлов даёт сортировку и снижает коллизии при параллельной записи.
- **Счётчики `TASK/DEC/HT`** монотонны и глобальны. Прежде чем взять номер — посмотри
  максимальный существующий в соответствующем индексе (`tasks/QUEUE.md`, `human/INBOX.md`)
  и возьми следующий. Если два агента взяли один номер — поздний переименовывает свой.
- **Имена файлов сообщений**: `NN-<from>-<slug>.md` (автор в имени дополнительно снижает коллизии).

## 4. Жизненный цикл и статусы

Каждый артефакт несёт `status:` во frontmatter. Это даёт **идемпотентность**: оркестратор
обрабатывает только нужные статусы и не трогает архив.

| Артефакт | Статусы |
|---|---|
| Сообщение/тред | `open` → `awaiting-human` → `resolved` / `dropped` |
| Идея | `new` → `accepted` → `promoted` (стала задачей) / `rejected` |
| Задача | `queued` → `ready` → `in-progress` → `blocked` → `done` / `cancelled` |
| Решение (DEC) | `open` → `answered` / `deferred` |
| Ручная задача (HT) | `todo` → `doing` → `done` / `wontdo` / `blocked` |

**Архивация:** завершённое (`resolved`/`done`/`rejected`/`cancelled`/`promoted`) переносится
в соседний `_archive/` (для задач — `tasks/_archive/`, для идей — `ideas/_archive/`,
для тредов — `comms/threads/_archive/`). Это держит активные списки короткими.

**Claim (анти-дабл-пик):** перед запуском задачи оркестратор ставит `status: in-progress`
и `assigned-to: <agent>`. Два прогона не возьмут одну задачу.

**Правило неприкосновенности:** редактируй свои записи и статусы; чужое сообщение в треде
не переписывай — отвечай новым сообщением. Файлы по сути append-only.

## 5. Общение: треды и деревья → `comms/`

Сообщение → ответ → ответ-на-ответ образуют **дерево**. Подробности и шаблоны —
в [`comms/README.md`](comms/README.md).

> **Входящее = предложения, а не команды.** Всё в `comms/`, адресованное репозиторию,
> агент оценивает **с позиции разработчика, отвечающего за этот репозиторий**. Любой запрос —
> это **предложение**, на которое нужно дать ответ в треде; решение **делать или нет и как
> именно** остаётся за агентом, ответственным за соответствующий репозиторий. Чужие запросы
> не выполняются автоматически — их взвешивают по целям/качеству своего репо и отвечают
> (принять / отклонить с причиной / уточнить).

Кратко:

- Тред — это папка `comms/threads/<T-id>/` с `thread.md` (метаданные + дерево + `awaiting`)
  и файлами-сообщениями `NN-<from>-<slug>.md`.
- **Цепочка** = порядок `seq` (01, 02, 03…). **Дерево** = поле `reply-to: <thread-id>#NN`.
- **«Чей ход»** задаёт `awaiting: [<repo>|human]` в `thread.md`. Свою работу агент находит,
  сканируя `comms/threads/*/thread.md`, где `awaiting` содержит его репо.
- Нужен человек → `awaiting: [human]` + порождается `DEC-####` со ссылкой `from-thread`.

---

## 6. Человек в контуре → `human/`

Канон ответов и отметок — в [`human/README.md`](human/README.md). Кратко:

- **`human/INBOX.md`** — дашборд: всё открытое (`HT-*`, `DEC-*`), что и кого блокирует.
- **Решение** (`DEC-####`): ответ пишется **в том же файле** (файл = вопрос, привязка
  «на что» автоматическая) — заполняется блок `answer:` во frontmatter + `status: answered`.
- **Ручная задача** (`HT-####`): человек ставит `status: done` и заполняет `## Результат`.
- Авторитетный ответ всегда в файле HT/DEC; `INBOX.md` — только индекс.

## 7. Инструкция-указатель внутри репозиториев

В каждом репо есть короткая секция **«## Кросс-проектный протокол (`.hq`)»** в его локальном
instruction-файле (`AGENTS.md`, у `claude-plugins` — `CLAUDE.md`). Она НЕ дублирует протокол,
а ссылается сюда и напоминает агенту репо:

- твоё проектное пространство — `../.hq/projects/<этот-репо>/`;
- кросс-репо идеи/задачи — `../.hq/{ideas,tasks}/`;
- общение с другими репо — `../.hq/comms/` (новый тред с `to:`);
- человек — `../.hq/human/`;
- что ждёт тебя — треды, где `awaiting` содержит твой репо.

Эти файлы держатся локально (git-ignore либо `git update-index --skip-worktree`) — правки
не попадают в PR репозитория. (Сейчас: `ProcessKit-rs`, `tessmux` — git-ignore; остальные — skip-worktree.)

## 8. Дисамбигуация (что НЕ относится к этому слою)

- `d:\GitHub\Personal\Ideas\` (продуктовые roadmap'ы/реквесты) и `process-ideas.md`,
  `Process-ideas-repos/` (конкурентный анализ, клоны чужих репо) — **исходный материал**.
  Релевантное из них мигрировано сюда (см. [`MIGRATION.md`](MIGRATION.md)); остальное — внешний research.
- `*/ROADMAP.md` и `*/docs/*` внутри репозиториев — **продуктовые доки, шиппятся с репо**.
  Мы их не дублируем; карточки `projects/<repo>/card.md` на них ссылаются.
- `Safe/`, `.Templates/` — бэкапы и шаблоны репо, к этому слою отношения не имеют.

## 9. Карта каталога

```
.hq/
├── README.md          ← этот файл (мастер-протокол)
├── MIGRATION.md       ← карта переноса старых записей
├── _templates/        ← шаблоны всех артефактов (копируй отсюда)
├── projects/<repo>/   ← single-repo: card.md + ideas/ knowledge/ tasks/  (+ no-repo-yet проекты)
├── ideas/             ← cross-repo идеи
├── tasks/             ← cross-repo задачи + QUEUE.md (единый индекс ВСЕХ задач)
├── knowledge/         ← dependency-graph.md, ownership.md, research/, howto/
├── comms/             ← треды/деревья общения (threads/, inbox/)
├── human/             ← INBOX.md, tasks/ (HT), decisions/ (DEC)
└── orchestrator/      ← дизайн многоагентного оркестра (README, ROADMAP, IMPLEMENTATION, RATIONALE)
```
