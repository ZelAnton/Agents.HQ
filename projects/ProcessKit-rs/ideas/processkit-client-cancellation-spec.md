# ProcessKit-rs — Client-Level Cancellation Requirements (from vcs-toolkit-rs)

A requirements note from the `vcs-toolkit-rs` project (the `vcs-git` / `vcs-jj` /
`vcs-github` / `vcs-gitlab` / `vcs-gitea` / `vcs-core` / `vcs-forge` crates,
which build on `processkit`'s `CliClient`). Same contract as the earlier
`Error::Exit.stdout` requirement (shipped in 0.5.0) and the streaming-hardening
note (`processkit-streaming-spec.md`): vcs-toolkit does **not** fork or vendor
processkit — needs are raised here as behavioural requirements, processkit
decides the implementation.

**Status.** Requirements, not commitments. Validated against processkit
**0.7.1** source (file:line references refer to the
`ProcessKit-rs-0.7.1` snapshot).

---

## Motivation

0.7 shipped cancellation right: `Command::cancel_on(token)` (`command.rs:287`),
re-exported `CancellationToken` (`lib.rs:288`), cancellation-beats-timeout
(`command.rs:268`), retry treats `Cancelled` as terminal. vcs-toolkit wants to
adopt it — its consumers (vcs-flow-rs, agent-workspace, the `vcs-mcp` server)
are agent orchestrators where a controller legitimately cancels a stuck
long-runner (`gh run watch`, a fetch over a dead link) **without owning or
aborting the task that awaits it**.

**Why per-command `cancel_on` alone is not adoptable by the wrappers.** The
toolkit's typed methods build and consume their `Command` internally
(`self.core.command_in(dir, args)` → terminal); the `Command` never crosses the
public API. To thread a token per call, vcs-toolkit would have to either:

- put `CancellationToken` parameters into its **object-safe, mockall-mocked**
  `*Api` traits (couples every mock/consumer/`&dyn` to tokio-util types, and
  feature-gated trait methods fragment the API), or
- touch every command-construction site (~30 per wrapper crate) to plumb an
  ambient token.

Both were rejected. The natural fix lives in `CliClient`: it already owns the
"applies to every command" defaults — `default_timeout`, `default_env`,
`default_env_remove` (`client.rs:70-95`, applied in `apply_defaults`,
`client.rs:118`) — and the `cli_client!` macro re-emits exactly those builders
on every generated client (`client.rs:233-248`). Cancellation is the one
run-control default missing from that set.

**The payoff for the ecosystem:** with a client-level default, vcs-toolkit
needs **zero new public API**. Its facades already accept pre-configured
clients (`Repo::from_git(root, cwd, client)`, `Forge::for_github(cwd, client)`),
so a consumer writes:

```rust
let token = CancellationToken::new();
let gh = GitHub::with_runner(JobRunner::new()) // or GitHub::new()
    .default_cancel_on(token.child_token());
let forge = Forge::for_github(cwd, gh);
// ... controller cancels `token` → every in-flight gh command of THIS client
// dies (kill-on-close tree), surfacing Error::Cancelled to the awaiting call.
```

Per-operation scoping falls out of cheap client construction (one client per
cancellable scope), not per-call parameters.

---

## R1 — `CliClient::default_cancel_on` (primary)

**Today.** `Command::cancel_on` exists per command; `CliClient` has no
cancellation default (`grep cancel src/client.rs` → nothing).

**Requirement.** A client-level builder — working name
`default_cancel_on(token: CancellationToken) -> Self` — behind the existing
`cancellation` feature, stored on `CliClient` and applied to every command the
client builds (the `apply_defaults` path), with:

1. **Documented precedence** vs a per-command `cancel_on`. Either is
   acceptable, but it must be explicit:
   - *override* (per-command token replaces the default — consistent with how
     an explicit setting beats a default), or
   - *combine* (cancel when **either** fires — the child-token composition).
   vcs-toolkit has no hard preference; if *override* is chosen, a consumer can
   still compose manually via `child_token()`.
2. **`cli_client!` macro emission**: the macro must re-emit the builder
   (cfg-gated on the feature) like it does `default_timeout`/`default_env`, so
   `Git`/`GitHub`/… clients generated by downstream crates expose it without
   any wrapper-crate code.
3. Existing per-command semantics unchanged: cancellation is always
   `Error::Cancelled`, beats a concurrent timeout, kills the process tree.

**Acceptance.** A test in processkit: a client with `default_cancel_on(t)` runs
a hanging child; cancelling `t` resolves the call with `Error::Cancelled` and
the tree is dead. A second test pins the documented precedence/combination
against a per-command `cancel_on`.

## R2 — Hermetic testability of cancellation (doubles)

**Today.** `doubles.rs:13` documents the gap honestly: ScriptedRunner replies
synchronously and "a `cancel_on` token is not observed — script an
[error] instead to exercise cancel handling". That lets a consumer fake the
*aftermath* (`Error::Cancelled` as a canned reply) but not the *behaviour*
(a call that genuinely blocks until cancelled) — so a consumer's cancellable
path (does my orchestration actually cancel? does it clean up?) is untestable
without a real binary.

**Requirement.** A doubles story for cancellation, e.g. a `Reply::pending()`
(or `Reply::until_cancelled()`) that makes `ScriptedRunner` park the call until
the command's token (per-command or client default) fires, then resolve with
`Error::Cancelled` — mirroring the live contract. `RecordingRunner`
pass-through should keep recording the invocation.

**Acceptance.**
`ScriptedRunner::new().on(["run", "watch"], Reply::pending())` + a client with
`default_cancel_on(t)`: the awaited call does not resolve until `t.cancel()`,
then yields `Error::Cancelled` naming the program; the invocation is recorded.

## R3 (secondary, optional) — generic per-command customizer

`default_cancel_on` is the third hand-rolled "apply to every command" need
vcs-toolkit has hit, after env policy (it wraps commands in `c_locale`/
`no_editor`/`GIT_OPTIONAL_LOCKS` helpers today) and the streaming-spec's R4
(`on_command` hook). A single generic
`CliClient::default_map(impl Fn(Command) -> Command + …)` would subsume all
three. It ranks **strictly below R1**: a typed, narrow default is easier to
reason about, introspect, and document than a stored closure — if processkit
prefers to add only R1, that fully satisfies this note. (If R3 is ever built,
fold the streaming-spec's R4 into it rather than shipping both.)

## Consumer sketch (what vcs-toolkit will do once R1 lands)

Nothing, structurally — that is the point. No new vcs-* API: the existing
`with_runner`/`from_git`/`for_github` constructors already accept
pre-configured clients, so cancellation arrives for consumers as
"configure your client" documentation plus a cookbook recipe:

```rust
// vcs-flow / agent-workspace / vcs-mcp side:
let token = CancellationToken::new();
let repo = Repo::from_git(root, cwd, Git::hardened().default_cancel_on(token.child_token()));
tokio::spawn(watchdog(token.clone()));           // external controller
repo.fetch().await                               // Err(Vcs(Cancelled)) if cancelled
```

vcs-toolkit will add (gated on the processkit release that ships R1+R2): a
hermetic test that a cancelled client surfaces `Error::Cancelled` through the
facade classifiers as "not transient / not a conflict" (the classifier
catch-all already guarantees it — the test pins it), and a cookbook entry.

## Compatibility

- Additive only; everything behind the existing `cancellation` feature. No
  breaking changes requested.
- MSRV: stay ≤ 1.88 (the floor shared by processkit 0.7.1 and vcs-toolkit).
- vcs-toolkit pins adoption via `[workspace.dependencies] processkit = "0.7.x"`
  once the shipping version is known.

## Response (processkit, 2026-06-07)

**R1 — ACCEPTED, implemented.** `CliClient::default_cancel_on(token)` behind
the `cancellation` feature, applied in `apply_defaults` alongside
`default_timeout`/`default_env`. **Precedence chosen: override** — the default
is applied when the command is built, so a per-command `cancel_on` chained on
the returned command *replaces* it (explicit beats default, exactly like a
per-command `timeout` after `default_timeout`); compose via `child_token()`
when both must fire. `cli_client!` re-emits the builder on generated wrappers
(via an internal feature-selected helper macro, so the cfg is evaluated
against processkit's features, not the downstream crate's). Both acceptance
tests shipped: a real hanging child cancelled through a client default
(integration), and the precedence pin (hermetic).

**R2 — ACCEPTED, implemented.** `Reply::pending()` (`cancellation` feature):
`ScriptedRunner` parks the matched call until the command's token — per-command
or client default — fires, then resolves `Err(Error::Cancelled { program })`.
`RecordingRunner` records the invocation before delegating (unchanged). A
pending reply for a command with **no** token parks forever (documented:
behaves like a hung child nobody can cancel). The spec's acceptance test is in
processkit's suite verbatim (`run watch` + `default_cancel_on`).

**R3 — DECLINED.** A typed, narrow default wins over a stored
`Fn(Command) -> Command` closure for introspection (`Debug`), documentation,
and cassette/recording transparency; R1 covers the stated need. Revisit only
if a *third* typed "apply to every command" candidate accumulates — then fold
the streaming-spec's R4 into one design rather than shipping either piecemeal.

**Ships in:** the next processkit release (will be **0.8.0** — the release
also carries an unrelated breaking change, `SupervisionOutcome` becoming
`#[non_exhaustive]`). MSRV unchanged (1.88). Pin `processkit = "0.8"` with
`features = ["cancellation"]`.
