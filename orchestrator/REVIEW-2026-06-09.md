# Ревью оркестратора `.hq` — состязательное многоагентное (2026-06-09)

**Метод:** workflow `orchestrator-adversarial-review` — 6 независимых линз (надёжность/безопасность,
простота/осуществимость, корректность слияния, механика Claude Code, согласованность спек/схем,
полнота) → независимая проверка каждой находки скептиком (Sonnet) → синтез.
**Объём:** 43 агента, 36 находок, 34 подтверждено → **18 уникальных**
(high 0, medium 11, low 7).

> Источник: фоновый прогон `wv4qnmeeh`. Отчёт сгенерирован синтезатором, сохранён как артефакт.

---
## Сводка

После дедупликации 30 исходных находок из 6 линз (reliability-safety, simplicity-feasibility, merge-correctness, cc-mechanics, spec-consistency, completeness-gaps) свёрнуты в **18 уникальных проблем**. Severity берётся из подтверждённых вердиктов (а не из исходных оценок линз — многие были снижены с учётом фазовости).

**Распределение по подтверждённой severity:** high — 0, medium — 11, low — 7, всего — 18.

Ни одна находка не подтверждена как high: фазовость P0/P1 (propose-only / dry-run, ручной запуск, человек ревьюит перед `--apply`, единственный реальный тред в skip-list) — реальный и работающий митигатор, который последовательно снижает severity. Однако это не отменяет проблем: значительная их часть — это **архитектурные дыры, которые нужно закрыть ДО включения соответствующей фазы** (P2 — claim/lease, P4 — авто-land/risk(), P5 — intra-repo параллелизм), а не «доделать постфактум».

**Ключевые сквозные темы:**
1. **Расхождение спецификация ↔ реализация ↔ данные.** SKILL.md обещает логику (фолбэк адресата, валидацию графа, проверку seed, retry), которую headless `comms.ps1` не реализует. При этом оба заявлены как «те же спецы/схемы».
2. **Детерминированные функции отданы LLM.** Аллокация монотонных TASK-#### и проверка parallel-safe возложены на LLM-планировщик, что прямо нарушает принцип №2 дизайна («скучную логику — программе»).
3. **Извлечение адресата по `^to:`** — самая частая находка (4 линзы): поля `to:` нет ни в шаблоне, ни в реальных тредах; источник правды — `awaiting`.
4. **Предохранители необратимых шагов объявлены, но не специфицированы:** lock, `risk()` для auto-land, гейт «нет нерешённых конфликтов», атомарность кросс-репо land — существуют как намерения без исполнимых определений.
5. **Идемпотентность и контур человек-в-петле (DEC) не замкнуты** — повторный тик переоценивает тред заново, ответ DEC не возвращается в обработку.

## Находки (по убыванию severity)

### MEDIUM

#### M1. Извлечение репо-адресата по `^to:` ломается на реальных и фикстурных тредах (объединено: reliability-safety + simplicity-feasibility + cc-mechanics + spec-consistency)
- **Проблема:** `comms.ps1:83` определяет адресата регэкспом `^to:\s*(\S+)` по конкатенации всех *.md треда, иначе `unknown`. Но поля `to:` нет ни в `_templates/thread.md`, ни в реальном треде, ни в фикстуре — источник правды «чей ход» по протоколу это `awaiting`. SKILL.md шаг 2 предписывает фолбэк `to:` → `awaiting` → `participants`, но скрипт его не реализует. Дополнительно: при многосообщенческом треде берётся ПЕРВОЕ совпадение `^to:`, и корректность держится на случайном алфавитном порядке `Get-ChildItem`; `awaiting` может содержать несколько репо — не определено, кому адресовать.
- **Почему важно:** Тихий отказ — триаж отрабатывает «с позиции владельца `unknown`», читает несуществующую `projects/unknown/card.md`, ломается ключевой принцип «входящее = предложения, оцени с позиции владельца». Воспроизводится уже на фикстуре; сломается на первом же новом реальном треде без `to:`.
- **Фикс:** Извлекать адресата из frontmatter `thread.md` с фолбэком `awaiting` (первый репо ≠ human) → `scope: single:<repo>` → `to:` → `participants`; читать `thread.md` отдельно от тел сообщений ДО join. При `unknown` — не звать triage, а писать в `tick.errors` и пропускать. Для `awaiting` с несколькими репо — зафиксировать правило (triage на каждого / первый = primary) синхронно в SKILL.md и hq-triage.md. Добавить фикстуру БЕЗ `to:`.
- **Фаза/файл:** P1 — `.hq/orchestrator/bin/comms.ps1:83`, `skills/comms/SKILL.md` шаг 2, `comms/README.md`.

#### M2. Гонка за номера TASK-#### — аллокация отдана LLM-планировщику без резервирования (объединено: simplicity-feasibility + spec-consistency)
- **Проблема:** `hq-planner.md` правило 7 предписывает LLM брать «следующие свободные TASK-####» из QUEUE.md. `comms.ps1` в одном прогоне вызывает planner независимо для каждого accept-seed, передавая один и тот же неизменный QUEUE.md (в dry-run между вызовами ничего не пишется). При ≥2 accept оба planner-а назначат пересекающийся диапазон → коллизия id. Симптом уже задокументирован: фикстура заняла 0005–0010, и QUEUE правили вручную примечанием «в очередь НЕ входят».
- **Почему важно:** Распределение монотонных id — детерминированная функция, которую дизайн прямо запрещает отдавать LLM. Дубли TASK-#### ломают `depends_on`/waves/единый индекс. Контракт уже зашит в схему и спеку агента — чинить позже дороже.
- **Фикс:** Planner возвращает локальные плейсхолдеры (`T1`,`T2`…) и рёбра между ними; Дирижёр под локом детерминированно присваивает финальные TASK-#### сквозным счётчиком по ВСЕМ seed прогона и переписывает `depends_on`/`parallel_safe_with`/waves по мапе. Обновить `planner.schema.json` и `hq-planner.md`.
- **Фаза/файл:** P1 (контракт) → проявится в P3+ — `hq-planner.md` правило 7, `schemas/planner.schema.json`, `comms.ps1` цикл foreach, `SKILL.md` шаг 3.

#### M3. Хрупкий парсинг вывода claude: `2>&1 | Out-String` + `ConvertFrom-Json` (объединено: simplicity-feasibility + cc-mechanics)
- **Проблема:** `comms.ps1:48` сливает stderr в stdout (`2>&1`) и парсит весь блоб `ConvertFrom-Json`. Любой шум на stderr (баннер обновления, warning MCP/permissions, прогресс) сделает блоб невалидным; `catch {}` молча проглатывает, `$env=$null`, функция падает в `throw` — штатный успех классифицируется как сбой и пишется в `tick.errors`. Прогон `fixture-20260609` создан вручную (нет каталога `plan/`, который скрипт создаёт; время округлено `19:30:00`) — то есть скрипт ни разу реально не запускался, и DoD TASK-0012 «`claude -p --json-schema` подтверждён» для headless-пути не обоснован.
- **Почему важно:** Это основной фолбэк-механизм P1, на который опирается весь crawl→run.
- **Фикс:** Не сливать stderr в парсимый поток — собрать stderr отдельно (`2>$log`) или через раздельные потоки; парсить только stdout; робастно извлекать JSON (от первой `{` до последней `}` или строку `{"type":"result"`). Различать «не запустился» и «модель не дала structured_output». Реально прогнать на фикстуре и зафиксировать результат в `_runs/`.
- **Фаза/файл:** P1 — `comms.ps1` Invoke-Specialist, строки 41–52.

#### M4. Заявленная валидация (JSON-схема + граф) не реализована в comms.ps1 (объединено: spec-consistency + completeness-gaps)
- **Проблема:** SKILL.md шаги 2–3 обещают проверку обязательных полей, `decision`∈enum, инварианта accept⇒seed, ацикличности `depends_on`, валидности ссылок и `parallel_safe_with` только при непересечении scope_paths. `comms.ps1` (строки 49–99) делает только `ConvertFrom-Json` + `if ($tri.reply_md)` + `if accept and $tri.seed`. Нет валидации против схем, нет проверки графа. `planner.schema.json` структурно неспособна выразить циклы/висячие ссылки. Плюс: при невалидном JSON SKILL предписывает «повтори один раз, потом пропусти» — `comms.ps1` НЕ ретраит (только пишет в errors).
- **Почему важно:** Мусорный/полу-валидный вывод (цикл в графе, висячая ссылка, parallel-safe поверх пересекающихся областей) пройдёт молча: в dry-run попадёт в `proposed-tasks`, при `--apply` — в QUEUE.md, после чего ready-set/волны на P2+ зациклятся. ROADMAP называет «качество триажа/плана» главным риском именно P1.
- **Фикс:** Реализовать в `comms.ps1` пост-валидацию: `Test-Json -Schema` (PS7), enum-поля, инвариант accept⇒seed, графовые правила (топосорт/ацикличность, разрешимость `depends_on`, непересечение scope для `parallel_safe_with`). Привести retry в соответствие с SKILL. Добавить негативную фикстуру (план с циклом) в DoD P1.
- **Фаза/файл:** P1 — `comms.ps1` строки ~49–99, `SKILL.md` шаги 2–3, `planner.schema.json`.

#### M5. Single-writer/lock из дизайна не реализован в comms.ps1 (reliability-safety)
- **Проблема:** Дизайн объявляет лок `.hq/orchestrator/.lock` краеугольным камнем против гонок (README §9, STATE.md, IMPLEMENTATION §2 — первая строка `tick()`), `.gitignore` уже содержит `orchestrator/.lock`. Но `comms.ps1` нигде не берёт лок, не пишет `.lock`, не проверяет стейл-лок, не имеет `finally`. `run_id` формата `yyyyMMdd-HHmm` — два запуска в одну минуту получат одинаковый id и перетрут `_runs/`.
- **Почему важно:** Лок должен существовать ДО первого мутирующего шага. `--apply` уже описан в SKILL.md как часть P1 — внедрять мутации поверх несуществующего лока нельзя.
- **Фикс:** Реализовать лок до любого скана/записи: атомарно `[System.IO.File]::Open(...CreateNew...)`, записать PID+время, в `finally` снять; при старте проверять стейл-лок по TTL и живости PID; гейтить `--apply` за локом. В `run_id` добавить секунды/PID.
- **Фаза/файл:** P0/P1 — `comms.ps1` (вся функция tick/main), `STATE.md`, `IMPLEMENTATION.md §2`.

#### M6. risk() для auto-land/push в main не определён — нет fail-closed дефолта (reliability-safety)
- **Проблема:** Самый необратимый шаг — авто-`jj bookmark move main` + `jj git push -b main` — гейтится `autonomy(repo)>=auto-low AND risk(integ)==low`, но функция `risk()` нигде не определена операционально. RATIONALE помечает риск-классификацию как открытый вопрос «калибровать на практике»; DoD TASK-0015 требует лишь «волна низкого риска авто-приземлена»; Verifier на P4 ещё не написан.
- **Почему важно:** В момент включения P4 «не сумел классифицировать» легко превращается в «считаю низким» и авто-пушит в main. Фазовость покрывает «когда», но не «что именно считать низким риском».
- **Фикс:** До реализации P4 зафиксировать исполнимое fail-closed определение low-risk: allow-list (нет изменений публичного API/CI/релизных файлов/секретов/миграций, diff ≤ N строк/файлов, тесты зелёные, Verifier=pass); любое НЕ-совпадение или неопределённость ⇒ DEC. Отсутствие/ошибка Verifier = высокий риск.
- **Фаза/файл:** P4 — `IMPLEMENTATION.md §2/§7`, `RATIONALE.md`, `tasks/TASK-0015`.

#### M7. Авто-разрешение jj-конфликтов гейтится только тестами — потеря намерения проходит в auto-land (merge-correctness)
- **Проблема:** Merge-агент «сохраняет намерение обеих сторон», но единственный машинный гейт после resolve — `tests_green`; при auto-low и `risk==low` уезжает в main. Тесты не ловят молчаливый выбор одной стороны (часто компилируется и проходит существующие тесты). Verifier не получает три версии (base/ours/theirs) и не обязан аудировать конфликтные хунки.
- **Почему важно:** «Сохранить намерение обеих» — самое error-prone суждение LLM; при auto-low уедет в main без человека.
- **Фикс:** Непустой `conflicts_resolved` ⇒ риск автоматически НЕ-low ⇒ land требует DEC (до накопления статистики). Передавать Verifier три версии + resolution и требовать явной проверки присутствия обеих сторон. Зафиксировать правилом в §4/§7.
- **Фаза/файл:** P4 — `IMPLEMENTATION.md §1.4/§2`, `README.md §9`.

#### M8. Кросс-репо: VCS-конфликт спутан с семантической несовместимостью; нет атомарного кросс-репо land (merge-correctness)
- **Проблема:** Заявлено «между репозиториями конфликтов нет в принципе». `land(repo, integ)` пер-репо; `depends_on` обеспечивает порядок ИСПОЛНЕНИЯ, но не блокирует land фундамента. Фундамент (ProcessKit-rs) может авто-land-нуться, пока потребитель (vcs-toolkit-rs) ещё в работе — main потребителя ломается на следующем fetch. Гейт проверяет `integ[repo]` в изоляции, без совместной сборки пары.
- **Почему важно:** Отсутствие текстового конфликта ≠ безопасность слияния; нарушается порядок из dependency-graph на этапе ПРИЗЕМЛЕНИЯ.
- **Фикс:** Запретить авто-land репо с незавершёнными downstream-задачами в текущем плане ⇒ DEC. Ввести «кросс-репо интеграционное множество»: land фундамента блокируется до зелёного гейта всех зависимых потребителей (совместная сборка против integ фундамента). Зафиксировать в §4 и риск-таблице §7; исправить вводящую в заблуждение фразу в §7.
- **Фаза/файл:** P4 — `README.md §7/§3`, `IMPLEMENTATION.md §2/§4`.

#### M9. Нет явного гейта «нет нерешённых jj-конфликтов» перед land (merge-correctness)
- **Проблема:** jj НЕ блокирует конфликт — материализует маркеры в файлы. Псевдокод после `agent(merge)` проверяет только `tests_green`; нет отдельного детерминированного предусловия land «`jj resolve --list` пуст». Если Merge вернул `needs_human` или тесты «зелёные» (конфликтная цель не компилируется/skipped), конфликтная ревизия может пройти к land. `executor-result.schema.json` допускает `tests/build = skipped`, и не сказано, что skipped блокирует land.
- **Почему важно:** Синтаксически возможен путь к land конфликтной ревизии независимо от тестов.
- **Фикс:** Добавить в §2 жёсткое предусловие `assert(no_conflicts(integ[repo]))` через `jj log -r 'conflicts()'`; иначе → escalated/DEC. Трактовать `build==skipped`/`tests==skipped` как НЕ-зелёный гейт.
- **Фаза/файл:** P4 — `IMPLEMENTATION.md §2/§4`, `executor-result.schema.json`.

#### M10. Lease/claim без TTL-политики и идемпотентного применения тика (reliability-safety)
- **Проблема:** (1) TTL лизы не задан численно и не связан с таймаутом исполнителя — при `lease < timeout` задачу переподхватят, пока первый жив (двойное исполнение); (2) claim хранит `assigned-to`, но не PID/host — нечем проверить смерть процесса; (3) `--apply` пишет тред+QUEUE+спеку+STATUS неатомарно, без пошагового журнала applied — краш посередине оставляет частично применённый тик.
- **Почему важно:** Полное лечение разумно отложено на P6, НО два инварианта обязаны быть зафиксированы сейчас, иначе P2/P3 унаследуют дыру, и recovery-логика P6 будет строиться на данных, которым нельзя доверять.
- **Фикс:** В STATE.md/IMPLEMENTATION зафиксировать: (a) `lease-until = claimed-at + executor_timeout + grace`, переподхват только после lease И подтверждённой смерти процесса; (b) добавить в claim `owner-pid`/`owner-host`; (c) пошагово журналируемый идемпотентный `--apply` (флаг applied на каждую мутацию в tick.json).
- **Фаза/файл:** P2/P3 (фиксация инвариантов) → P6 (реализация) — `STATE.md §claim`, `IMPLEMENTATION.md §6/§3`, TASK-0013/0014/0017.

#### M11. Защита от утечки секретов/локальных путей описана, но детектора нет; `_runs/` tracked, `.hq` имеет публичный remote (reliability-safety)
- **Проблема:** Контроль утечки целиком возложен на будущий Verifier (P4) — на P1 детектора нет. `.hq/.gitignore` не исключает `_runs/` (он tracked, `fixture-20260609/` уже в репо), у `.hq` публичный remote (`github.com/ZelAnton/Agents.HQ`). SKILL.md уже содержит абсолютные пути `d:/GitHub/Personal/` (строки 3,10,24,29,75) — уже в публичном репо. `--apply` пишет LLM-`reply_md` в tracked `comms/threads/`.
- **Почему важно:** Между P1 (есть генерация текста) и P4 (появляется проверка) — окно без детектора; канал утечки локальных путей/секретов в публичную историю реален уже на P1.
- **Фикс:** Дешёвый детерминированный pre-write скан (не LLM) уже в P1: regex на `d:/GitHub/Personal`, `C:\\`, `ghp_`, `xox`, `BEGIN PRIVATE KEY` по любому тексту в реальные файлы при `--apply` (и с P2 по diff исполнителя); совпадение ⇒ блок + эскалация. Исключить `_runs/` из git или явно решить вопрос трекинга.
- **Фаза/файл:** P1 — `comms.ps1`, `SKILL.md`, `.hq/.gitignore`, `IMPLEMENTATION.md §7`.

### LOW

- **Фаза/файл:** P1 — `.claude/skills/comms/SKILL.md` vs `.hq/orchestrator/skills/comms/SKILL.md`.

#### L2. Хрупкий skip-list (дублирован в двух местах) + отсутствие стратегии тестирования агентов (объединено: reliability-safety + completeness-gaps)
- **Проблема:** Запрет на тред `T-20260609-...` захардкожен дважды: `comms.ps1:22` и SKILL.md §«Жёсткие правила», без TTL и единого источника. STATE.md признаёт, что надёжная альтернатива (frontmatter `orchestrator: skip`) сознательно отвергнута. Параллельно: единственная синтетическая фикстура (только accept), нет golden-выходов, нет фикстур для reject/clarify/escalate, нет негативных кейсов (битый JSON, цикл в графе, пересечение scope).
- **Почему важно:** Два хардкода разъедутся (правят один, забывают второй) → вечный пропуск треда; переименование/новая точка входа (Rust-conductor P6, ручной `claude -p`) обходят оба. Единственная позитивная фикстура не даёт уверенности в «качестве триажа/плана» — главном риске P1.
- **Фикс:** Перенести признак пропуска в frontmatter треда (`orchestrator: skip` + причина) как единственный источник; хардкод оставить как аварийный фильтр. Завести набор фикстур на каждое решение triage + негативные, со сверкой выходов со схемами как часть exit-критерия каждой фазы.
- **Фаза/файл:** P1 (фикстуры) / до P3–P4 (механизм исключений) — `comms.ps1:22`, `SKILL.md`, `STATE.md`, `_fixtures/sample-inbound`.

#### L3. parallel-safe по непересечению scope_paths неполно: общие/моноширинные файлы (merge-correctness)
- **Проблема:** parallel-safe = «scope_paths не пересекаются И нет общего публичного API», но scope_paths по умолчанию каталоги, а «нет общего API/манифеста» — целиком суждение LLM, детерминированного гейта нет. Две задачи в одном репо почти всегда пишут в общие файлы вне «каталогов»: Cargo.toml/lock, CHANGELOG, lib.rs/mod.rs, Spring XML.
- **Почему важно:** Пара, помеченная parallel-safe, конфликтует или ломает потребителей общего API — «конфликтный шторм», который P5 обещает избежать.
- **Фикс:** Детерминированная проверка поверх LLM: (1) список «общих/моноширинных» файлов в card.md — любая трогающая их задача авто-сериализуется; (2) файловая гранулярность scope_paths для intra-repo; (3) поля `touches_public_api`/`touches_shared_manifest`, parallel-safe только при false у обеих.
- **Фаза/файл:** P5 — `IMPLEMENTATION.md §5`, `hq-planner.md` правило 4, `planner.schema.json`.

#### L4. parallel_safe_with не обязан быть симметричным; нет валидации взаимности (merge-correctness)
- **Проблема:** `parallel_safe_with` — свободный массив id; «безопасно параллельно» симметрично по смыслу, но схема и отбор батча этого не проверяют (подтверждено: TASK-0001↔0003 несимметричны в QUEUE.md). Двойной источник правды: явный `parallel_safe_with` от планировщика И вычисление непересечения scope_paths Дирижёром.
- **Почему важно:** Неопределённость, какой источник выигрывает; для intra-repo (P5) цена ошибки максимальна. (Смягчено тем, что Дирижёр фактически гейтит по scope_paths.)
- **Фикс:** Единственный авторитет — Дирижёр считает parallel-safe сам из scope_paths (+общие файлы из L3); `parallel_safe_with` — подсказка (AND, не OR). Добавить проверку симметрии и запрет одновременно `depends_on` и `parallel_safe_with` на одну задачу.
- **Фаза/файл:** P5 — `planner.schema.json`, `tasks/QUEUE.md`, `IMPLEMENTATION.md §2`.

#### L5. Перемещения/переименования файлов не покрыты scope_paths-моделью (merge-correctness)
- **Проблема:** scope_paths оперирует путями без различия edit/move/delete; `executor-result.schema.json` не имеет полей moved/renamed/deleted. Перемещение файла расширяет фактическую область на оба каталога и ссылки/импорты по репо — параллельная задача после rebase получит конфликт переименования.
- **Почему важно:** Подрывает базовое допущение «непересечение ⇒ чистое слияние». (Смягчено: актуально к P5; jj материализует конфликт явно, не тихо.)
- **Фикс:** Флаг `restructures: true` в spec/scope ⇒ всегда сериализуется (отдельная волна); добавить moved/renamed/deleted в executor-result; `out_of_scope_touched` ⇒ hard-stop для auto-land.
- **Фаза/файл:** P5 — `hq-planner.md` правило 2, `IMPLEMENTATION.md §4`, `executor-result.schema.json`.

#### L6. Слабый инвариант seed: if/then/else не гарантируется structured-output движком (объединено: simplicity-feasibility + cc-mechanics + spec-consistency)
- **Проблема:** Инвариант «accept⇒seed:object, иначе null» держится на `allOf/if-then-else` (triage.schema.json:35-41), который условно соблюдается LLM-движком. `comms.ps1:90` защищается частично (`accept -and $tri.seed`), не проверяет обратный случай и не сигналит ошибку при accept+null. Нет `minLength` на title/sketch (пустая строка валидна). Промпт hq-triage.md выражает инвариант одной фразой.
- **Почему важно:** При accept+null seed Дирижёр молча НЕ запланирует задачу — «accept без последствий». На P1 риск низкий (dry-run, ревью), но на P2+ (seed→planner→исполнение) мусорный seed дороже.
- **Фикс:** Явная пост-валидация после парсинга: `decision`∈enum; при accept — seed object с непустыми title/sketch и `repos.length≥1`; при не-accept — seed обязан быть null. Усилить схему `minLength`. Выделить инвариант отдельным правилом в hq-triage.md. Не размазывать инвариант по двум слоям.
- **Фаза/файл:** P1/P2 — `schemas/triage.schema.json`, `comms.ps1:90`, `hq-triage.md`.

#### L7. spec_md без frontmatter и неоднозначные пути `_templates/` (spec-consistency)
- **Проблема:** Шаблоны в `.hq/_templates/`, но SKILL.md (шаги 3,5) ссылается `_templates/task.md`/`reply.md` без корня — при базе `.hq/orchestrator/` такого каталога нет. `comms.ps1:96` пишет ТОЛЬКО `spec_md` без frontmatter (нет id/status/repos/depends-on); planner.schema.json этих полей не содержит. Файлы proposed-task несовместимы с `_templates/task.md` и немашиночитаемы по протоколу .hq.
- **Почему важно:** Рвёт цепочку schema(planner) → файл-спека → QUEUE; при `--apply` невалидные файлы попадут в `tasks/`; человек на ревью видит неполные артефакты.
- **Фикс:** Указать однозначный путь `.hq/_templates/...` в SKILL.md. Дирижёр генерирует frontmatter из planner-полей (id, repos, status:queued, priority, depends-on, parallel-safe-with, origin:<thread-id>) + тело spec_md; заменить голый `$task.spec_md | Set-Content` на сборку frontmatter+тело.
- **Фаза/файл:** P1 — `SKILL.md` шаги 3/5, `comms.ps1:96`, `.hq/_templates/`.

## Что ещё упущено (важные пробелы вне находок, но косвенно вытекающие из подтверждённых)

1. **Идемпотентность повторного тика (была отдельной находкой completeness-gaps, medium).** Отбор входящего детерминирован только полем `awaiting`; после `--apply` шаг 5 ставит `awaiting` на инициатора (тоже репо), статус остаётся `open` — тот же тред снова попадёт в скан и заново триажится. В dry-run каждый прогон триажит всё с нуля, повторный accept плодит те же TASK-#### (т.к. QUEUE в dry-run не пишется). Нет маркера «обработано до seq NN». **Фикс:** записывать `last-triaged-seq`; брать тред только при `max(seq) > last-triaged-seq`; planner — из персистентного счётчика. Фаза P1, файл `comms.ps1` Get-Inbound, SKILL.md шаги 1/5.

2. **Контур человек-в-петле (DEC) не замкнут (была находкой completeness-gaps, medium).** Цикл `escalate → DEC → ответ человека → возобновление` заявлен в README §6, SKILL.md шаг 1, IMPLEMENTATION §1.1, но: `comms.ps1` не сканирует `human/decisions` вовсе; нет схемы `dec-answer` с `from-thread`; не определено, как связать DEC обратно с тредом и переставить `awaiting`; нет маркера `consumed`. Escalate сейчас — тупик. P4 опирается на этот путь. **Фикс:** схема dec-answer, правило ре-входа в Дирижёре, поле `consumed-at`.

3. **Метрики перечислены, но не определены (была находкой completeness-gaps, medium).** IMPLEMENTATION §8 перечисляет метрики (% зелёных тестов, % конфликтов, % эскалаций, % откатов, доля авто-land, время задачи, токены/тик) без формул, окон и полей. `tick-log.schema.json` не содержит ни одного метрического поля. Сквозной гейт ROADMAP «не повышаем автономию, пока метрики предыдущей фазы не зелёные» неизмерим. **Фикс:** зафиксировать определения (числитель/знаменатель/окно), расширить tick-log.schema полями, собирать с P1/P2, а не вводить в P6.

4. **Дрейф путей `_runs`: дизайн обещает `tasks/_runs/`, реализация пишет в `orchestrator/_runs/` (была находкой simplicity, low).** Четыре дизайн-дока (README:254, IMPLEMENTATION:59/123, ROADMAP:36, заголовок STATE.md) указывают `tasks/_runs/`, а SKILL.md:29 и comms.ps1:32 пишут в `orchestrator/_runs/`. STATE.md самопротиворечив. P6-recovery будет искать журнал не там. **Фикс:** выбрать одно (логичнее `orchestrator/_runs/`) и синхронно привести 4 дока.

5. **Вводящий в заблуждение TODO-комментарий в comms.ps1 про `--json-schema` (была находкой cc-mechanics, low).** Код корректен (`--json-schema` принимает inline-строку), но комментарий предлагает «починку» (передать путь `$SchemaFile`), которая СЛОМАЕТ рабочий вызов. Ловушка для будущего сопровождающего. **Фикс:** удалить/переписать комментарий: «принимает inline JSON-строку, путь к файлу НЕ поддерживается»; добавить лёгкую `$schema | ConvertFrom-Json` валидацию.

6. **Системное наблюдение:** большинство дыр предохранителей (lock, risk(), гейт конфликтов, атомарный land, lease-инварианты) — это случаи «контракт объявлен в дизайне, но не имеет исполнимого определения или fail-closed дефолта». Стоит ввести единое правило проекта: **любой предохранитель необратимого/мутирующего шага должен иметь исполнимое определение и fail-closed поведение ДО включения фазы, которая его использует**, и явный чек этого в DoD соответствующей фазы.

## Что сделано верно (подтверждённо хорошо)

- **Фазовость как реальный митигатор.** Propose-only/dry-run по умолчанию, ручной запуск, `--apply` отдельным осознанным шагом, человек ревьюит перед мутацией, `autonomy=propose` по умолчанию, land за человеком до P4 — это последовательно и реально снижает severity почти каждой находки. Дизайн честно откладывает intra-repo параллелизм (P5) и hardening (P6) именно потому, что они опаснее.
- **Чтение `structured_output` корректно.** Путь `$env.structured_output` срабатывает первым при наличии схемы (это правильно), `result=''` при схеме обработан верно.
- **Код `--json-schema $schema` корректен** — флаг действительно принимает inline JSON-строку (проверено на claude 2.1.169); проблема только в вводящем в заблуждение комментарии.
- **Фильтр Get-Inbound уже использует `awaiting`** для отбора тредов (строка 69) — то есть правильный источник правды в коде уже присутствует, его просто нужно переиспользовать и для извлечения адресата (M1).
- **Частичная защита инварианта seed** в коде (`accept -and $tri.seed`) — надёжнее, чем if/then в схеме; направление верное, нужно лишь сделать её единственным источником и расширить.
- **Headless-путь правильно обходит «субагент не спавнит субагентов»** — `comms.ps1` зовёт отдельные процессы `claude -p`, а не вложенные Task (в отличие от интерактивного SKILL, которому нужна явная граница применимости «только верхний уровень»).
- **Консервативный дефолт планировщика** («сомнение → НЕ метить parallel_safe_with, ставить depends_on») — здравая стратегия, реально снижающая риск ошибочной параллелизации.
- **Риски частично осознаны в самом дизайне:** IMPLEMENTATION §7 уже фиксирует «конфликтные штормы (внутри-репо ∥)», README §2 — принцип «сначала параллелим между репозиториями»; `.gitignore` уже содержит `orchestrator/.lock` (инфраструктура под лок заявлена). Многие проблемы — это незавершённая реализация уже верно идентифицированных рисков, а не слепые пятна.
