# RATIONALE — как пришли к этому дизайну

Дизайн получен в несколько раундов **генерация → критика → оценка критики → синтез**. Здесь —
ход мысли и рассмотренные альтернативы, чтобы решения можно было оспорить осознанно.

---

## Раунд 1 — первый набросок

Шесть ролей-агентов (Triage, Planner, Scheduler, Executor, Merge, Supervisor), всё гоняется
периодически: scan → triage → plan → schedule → execute → integrate → mark. Исполнители — в
worktree, слияние — в конце.

## Раунд 1 — критика

1. **Конфликты заметены под ковёр.** Параллельные подзадачи в одном репо при пересечении дают
   merge-конфликты; «слить в конце» = большой взрыв. Самая сложная часть осталась нераскрытой.
2. **Автономный execute+merge+push опасен** для LLM-кода (особенно прямой push в main).
3. **Смешение детерминизма и LLM.** Если один большой LLM-агент и считает готовность, и пишет
   код, и сливает — это ненадёжно.
4. **Состояние и гонки** не продуманы: кто пишет очередь, что при крахе.
5. **jj не использован**, хотя репозитории на jj — а это меняет всю модель конфликтов.

## Раунд 2 — синтез после критики

- **Разделили плоскости:** детерминированный **Дирижёр** (управление) + **LLM-специалисты**
  (суждения). Чёткие узкие контракты с JSON-выходом.
- **Конфликты — структурно:** ключевое наблюдение — **между репозиториями конфликтов нет**.
  Значит, основной параллелизм — кросс-репо; внутри-репо сначала по очереди.
- **jj-native интеграция:** инкрементальный rebase на интеграционную ревизию; jj хранит
  конфликты не блокируя; чинит отдельный агент; гейт тестов после каждого шага.
- **Человек на ручке автономии** + гейты + откат + бюджеты.
- **Догфудинг стека** (agent-workspace, processkit, vcs-toolkit, tessmux, Claude Code headless).

## Раунд 2 — оценка критики (критика синтеза)

1. **Кастомный Rust-Дирижёр — преждевременно для v1.** Сначала достаточно скрипта/skill,
   зовущего `claude -p`. → ROADMAP начинается минимально (crawl→walk→run), Rust — фаза P6.
2. **Конфликт лучше предотвращать, чем разрешать.** → v1: один исполнитель на репо за тик,
   параллелизм только между репозиториями; внутри-репо параллелизм — поздняя фаза P5 с разбиением
   по областям и обязательной сериализацией при сомнении.
3. **Проверяемость — не опция.** → отдельный Verifier-гейт (переиспользует `code-review`/
   `security-review`) + DoD; тесты обязательны.
4. **Доверие наращивается рампой.** → диск автономии: `propose → assist → auto-low`, per-repo.
5. **Реалистичный старт уже возможен** на примитивах Claude Code (schedule/loop + skills +
   Agent + agent-workspace + jj/bash) — не нужно ждать большой системы.
6. **Восстановление после краха** через переоценку состояния из `.hq` + worktrees.

## Раунд 3 — финальная форма

ROADMAP P0→P6 (фундамент → триаж/план → 1 задача → кросс-репо ∥ → jj-интеграция+авто-land →
внутри-репо ∥ → hardening). Архитектура из 5 слоёв (данные/дирижёр/агенты/изоляция/наблюдаемость).
Контракты агентов, схемы состояния, jj-интеграция, риски/контроли, метрики — в `IMPLEMENTATION.md`.

## Ключевые решения и альтернативы

| Решение | Почему | Отвергнутая альтернатива |
|---|---|---|
| Детерминированный Дирижёр + LLM-специалисты | надёжное управление, умные суждения; легко тестировать | один «всемогущий» LLM-агент — непредсказуемо, плохо отлаживается |
| Кросс-репо параллелизм **сначала** | физически без конфликтов — быстрый безопасный выигрыш | сразу внутри-репо ∥ — конфликтные штормы на старте |
| jj-native инкрементальная интеграция | конфликты не блокируют, чинятся асинхронно, рано и мелко | git big-bang merge — блокировки, поздние крупные конфликты |
| Диск автономии + гейты + land за человеком до P4 | наращивание доверия, безопасность LLM-кода | полная автономия сразу — риск порчи remote |
| Скрипт/skill-Дирижёр в v1, Rust в P6 | быстро начать, дёшево менять | сразу Rust-бинарь — долго, преждевременная оптимизация |
| Единственный писатель состояния | нет гонок при многих исполнителях | свободная запись всеми — рассинхрон/порча QUEUE |
| `parallel-safe = false` по умолчанию | безопасность важнее скорости | оптимистичный параллелизм — конфликты |
| Догфудинг своего стека | оркестр = витрина и драйвер развития проектов | сторонние инструменты — не развивают экосистему, лишние зависимости |
| Данные = файлы `.hq` в git | прозрачно, версионируемо, откат, человекочитаемо | внешняя БД/очередь — непрозрачно, лишняя инфраструктура |

## Открытые вопросы (решить по ходу)
- Runtime LLM: Claude Code headless (`claude -p`) vs Agent SDK — начать с headless, при росте — SDK.
- Риск-классификация для `auto-low`: эвристика (объём diff, затронут ли публичный API/CI/релиз) +
  мнение Verifier; калибровать на практике.
- Гранулярность области (`scope_paths`): файлы vs каталоги vs модули — начать с каталогов.
- Нужен ли отдельный Reporter-агент или это делает Дирижёр шаблоном — пока Дирижёр.

## Что можно усилить дальше (если захотим глубже)
Состязательная проверка дизайна несколькими независимыми агентами (надёжность/безопасность,
простота/осуществимость, корректность слияния), турнир альтернативных планировщиков, или
прогон через многоагентный workflow — по запросу.
